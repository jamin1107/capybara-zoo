# 需求文档生成专家

作为测试专家，本技能将模糊、零散的需求转化为清晰、可测试、可落地的需求文档，为后续测试设计、开发落地提供明确依据。

## 核心原则

**"先明确边界、再拆解细节、后规范输出"**

## 依赖MCP工具

**必须配置 docx-generation MCP**，用于生成 .docx 格式的Word文档。

### MCP配置文件

在项目根目录 `.trae/mcp.json` 中添加以下配置：

```json
{
  "mcpServers": {
    "docx-generation": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-docx-generator"],
      "description": "生成 .docx 格式的 Microsoft Word 文档"
    }
  }
}
```

### docx-generation MCP 必需能力

MCP工具必须支持：
- 创建空白的 .docx 文档
- 添加多级标题（Heading 1/2/3）
- 添加表格（带表头样式）
- 添加项目符号列表
- 添加普通段落
- 设置单元格背景色
- 保存文档到指定路径

## 输入来源

- UI图/UI截图（位于 `桌面/test/素材文件夹`）

结构化需求文档，包含：
- 文档总则
- 整体功能概述
- 详细需求描述（功能模块→功能点→细节描述）
- 非功能需求
- 需求验收标准
- 附件与附录

## 第一步：需求收集与原始信息梳理

### UI图类需求梳理

逐屏截图分析，记录每一个可视化元素及潜在交互逻辑：

1. **页面布局**：组件位置、尺寸、对齐方式
2. **元素属性**：按钮、输入框、下拉框等的名称、默认状态、禁用/启用条件
3. **跳转逻辑**：点击某元素后跳转至哪个页面、返回路径
4. **异常展示**：空数据、错误提示的UI样式（未明确标注需记录疑问）
5. **适配要求**：不同屏幕尺寸、暗黑模式等（UI图未明确的需标注）

### 简单功能描述类需求梳理

提取核心功能关键词，拆解每一句话的功能意图：

1. **功能主体**：谁在使用该功能（用户、管理员）
2. **操作动作**：查询、提交、删除等
3. **输入输出**：输入什么内容、输出什么结果
4. **约束条件**：权限限制、格式要求等（未明确的需记录）
5. **关联功能**：与其他功能的交互

### 信息归档

按"页面/功能模块"分类记录，标注：
- ✅ 明确信息
- ⚠️ 模糊信息
- ❌ 缺失信息

---

## 第二步：需求澄清与边界确认

### 4个重点确认项

1. **功能边界**：明确"该功能做什么、不做什么"
   - 查询范围、查询频率限制
   - 登录方式、验证码校验

2. **交互逻辑**：补充未展示的交互细节
   - 加载状态、错误提示样式和位置
   - 页面刷新后数据是否保留
   - 弹窗关闭方式

3. **异常场景**：从测试视角出发主动询问
   - 网络中断时功能如何响应
   - 非法数据输入处理
   - 高频操作防重处理
   - 权限不足时的跳转/提示

4. **非功能需求**：明确性能、兼容性、安全性要求
   - 页面加载时间要求
   - 支持的浏览器版本
   - 防SQL注入、数据加密要求
   - 并发用户数限制

### 输出物

形成《需求澄清纪要》，由需求方确认签字（或线上确认）。

---

## 第三步：需求拆解与结构化梳理

### 3层拆解结构

#### 第1层：整体功能模块拆分

按业务流程或页面划分核心模块。

**示例**：
- 用户管理功能 → 用户列表查询、用户新增、用户编辑、用户删除

#### 第2层：模块内功能点拆解

每个功能点明确"输入、操作、输出、约束"。

**示例**（用户新增）：
| 要素 | 内容 |
|------|------|
| 输入 | 用户名（必填、长度6-20位、不允许特殊字符）、手机号（必填、11位有效手机号）、密码（必填、长度8-16位、含字母+数字） |
| 操作 | 点击"新增"按钮，提交输入内容 |
| 输出 | 新增成功（提示"新增用户成功"，跳转至用户列表）、新增失败（提示对应错误信息） |
| 约束 | 用户名不可重复、手机号不可重复、仅管理员可操作 |

#### 第3层：补充细节信息

- **UI细节**：元素位置、样式、默认值
- **交互细节**：点击、输入、跳转的具体逻辑
- **异常细节**：每种异常场景的处理方式
- **关联细节**：与其他模块的交互逻辑

---

## 第四步：规范撰写需求文档

### 标准结构

#### 1. 文档总则

```markdown
## 1. 文档总则

### 1.1 文档目的
明确本文档的用途（指导开发、测试落地，明确需求边界）

### 1.2 需求背景
简要说明该需求的业务场景、开发目的

### 1.3 适用范围
明确文档适用的人员（开发、测试、产品）、适用的功能模块

### 1.4 术语定义
明确文档中涉及的专业术语、缩写
```

#### 2. 整体功能概述

简要描述需求的核心功能、整体业务流程。

#### 3. 详细需求描述（核心部分）

按"功能模块→功能点→细节描述"结构撰写，每个功能点需明确：

| 要素 | 说明 |
|------|------|
| 功能描述 | 核心用途 |
| UI要求 | 元素布局、样式、默认状态、交互反馈 |
| 操作步骤 | 用户/管理员的操作流程 |
| 输入要求 | 必填/非必填、格式、长度、范围 |
| 输出结果 | 成功/失败的反馈 |
| 异常处理 | 每种异常场景的处理逻辑 |
| 关联功能 | 与其他模块的交互逻辑 |

#### 4. 非功能需求

| 类型 | 内容 |
|------|------|
| 性能需求 | 页面加载时间≤2秒，接口响应时间≤500ms，支持100人同时在线 |
| 兼容性需求 | Chrome 100+、Edge 100+、Windows 10+、iOS 15+、Android 11+ |
| 安全性需求 | 权限控制、数据加密、防SQL注入/XSS攻击 |
| 易用性需求 | 操作步骤不超过3步、错误提示明确 |

#### 5. 需求验收标准（测试核心依据）

将详细需求转化为可验证、可量化的验收条件。

**示例**：
- 验收条件1：点击"新增"按钮，成功弹出新增弹窗，弹窗布局与UI图一致
- 验收条件2：用户名输入为空时，点击"提交"，提示"用户名不能为空"，弹窗不关闭
- 验收条件3：输入符合要求的用户名、手机号、密码，点击"提交"，提示"新增成功"，跳转至用户列表
- 验收条件4：网络中断时，点击"提交"，提示"网络异常，请重试"，输入的内容不丢失

#### 6. 附件与附录

- UI图附件
- 需求澄清纪要
- 需求变更历史

---

## 第五步：需求文档校验与优化

### 5个校验维度

1. **校验可测试性**
   - 每个需求点、验收条件是否可验证
   - 避免"体验良好""操作流畅"等模糊表述
   - 替换为可量化标准：如"点击按钮后，响应时间≤300ms"

2. **校验无歧义**
   - 检查多义词、模糊表述
   - 确保不同人理解一致

3. **校验边界完整性**
   - 是否覆盖所有功能点、异常场景
   - 是否遗漏权限控制、防重处理等

4. **校验一致性**
   - UI描述与验收条件一致
   - 功能描述与关联功能一致

5. **需求方确认**
   - 提交给需求方确认所有内容符合预期

## 文档格式要求

**输出格式必须为 .docx（Microsoft Word文档）**，不得输出.md格式。

使用 python-docx 库生成Word文档，文档需包含：
- 标题层级（标题1/标题2/标题3）
- 表格（用于条件对照、功能点描述等）
- 项目符号列表
- 粗体/斜体强调

## 新增功能：需求文档评审分析

当用户上传 `.docx` 格式的需求文档时，执行以下流程：

### 流程概述

1. **解析文档** → 读取docx内容，提取文本和结构
2. **AI评审分析** → 根据5步评审法分析文档，按1,2,3,4...编号列出问题
3. **页面展示** → 将评审点展示在页面上，支持用户修改/新增/删除
4. **用户确认** → 用户确认评审点后，AI修改原文档
5. **生成标记文档** → 修改内容用**红色字体**高亮显示

---

### 第一阶段：文档解析与AI评审

#### 输入
- 用户上传 `.docx` 格式的需求文档

#### 输出
按以下5个维度，对需求文档进行评审，**按1,2,3,4...顺序编号**输出评审问题清单：

**评审点结构**：
```json
{
  "reviewPoints": [
    {
      "id": 1,
      "category": "功能边界模糊",
      "location": "章节3.2.1",
      "originalText": "原文内容...",
      "issue": "问题描述...",
      "suggestion": "修改建议..."
    }
  ]
}
```

#### 评审5维度（必须全部覆盖）

| 编号 | 评审维度 | 检查内容 |
|------|----------|----------|
| 1 | **功能边界** | 功能描述是否清晰、范围是否明确 |
| 2 | **交互逻辑** | 操作流程、跳转逻辑是否完整 |
| 3 | **异常场景** | 是否遗漏异常情况处理 |
| 4 | **非功能需求** | 性能、兼容性、安全性是否明确 |
| 5 | **验收标准** | 是否有可测试的验收条件 |

#### 评审问题分类

| 问题类型 | 标识 | 说明 |
|----------|------|------|
| ⚠️ 模糊信息 | 需明确 | 表述不清晰，需补充具体定义 |
| ❌ 缺失信息 | 需补充 | 缺少必要内容，如异常处理、约束条件 |
| 🔄 逻辑矛盾 | 需确认 | 前后描述不一致或逻辑冲突 |
| ⏳ 不完整 | 需完善 | 功能点、流程、场景不完整 |

---

### 第二阶段：页面交互

#### 评审点展示
- 按编号（1,2,3,4...）顺序展示所有评审点
- 每个评审点显示：分类、位置、原文、问题、建议
- 支持用户**新增**、**修改**、**删除**评审点

#### 操作约束
- 用户可以修改AI生成的评审点内容
- 用户可以手动新增评审点
- 用户可以删除不需要的评审点
- 点击"确认评审"按钮进入下一阶段

---

### 第三阶段：文档修改与生成

#### 输入
- 原需求文档（.docx）
- 用户确认的评审点列表

#### 处理逻辑
对每个评审点：
1. 根据`suggestion`修改原文`originalText`
2. 修改后的内容用**红色字体（#FF0000）**显示
3. 保留原文结构

#### 输出
- 生成新的 `.docx` 文档
- 修改的内容用**红色字体**高亮标注
- 文件命名：`原文件名_已评审.docx`
