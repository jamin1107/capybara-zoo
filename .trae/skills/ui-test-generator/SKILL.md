# UI 测试用例生成器

这个技能帮助测试工程师分析 UI 设计稿（原型图、截图、线框图）并生成全面的测试用例。通过从 UI 元素和交互中推断功能需求，解决需求不完整这个常见问题。

## 何时使用这个 Skill

**在以下情况下调用：**
- 用户提供了 UI 设计稿/图片并请求生成测试用例
- 用户想要分析 UI 设计进行测试
- 用户需要从 UI 设计中推导需求
- 用户问"怎么测试这个UI"或"为这个页面生成测试用例"

## 工作流程

### 步骤一：UI 分析与信息收集

1. **请求 UI 素材**
   - 请用户提供 UI 设计文件（PNG、JPG、Sketch、Figma 导出图等）
   - 请求任何现有的需求文档或规格说明
   - 询问背景：这是什么类型的应用？（Web、移动端、桌面端）

2. **分析 UI 元素**
   - 识别所有交互元素（按钮、输入框、菜单、链接）
   - 分析布局结构和视觉层级
   - 识别表单字段和验证要求
   - 注意导航模式和用户流程
   - 识别状态：默认、悬停、激活、禁用、错误、加载中

3. **推断业务逻辑**
   - 从字段类型/标签推断可能的数据验证规则
   - 从 UI 流程推断工作流程顺序
   - 从 UI 约束识别可能的边界情况
   - 分析错误处理和反馈机制

### 步骤二：需求澄清

**对于不完整/不明确的需求，主动询问：**
- 输入字段的预期数据类型和格式是什么？
- 验证规则是什么（长度、格式、必填）？
- UI 中可见的用户角色和权限是什么？
- 每个交互的预期行为是什么？
- 错误状态和错误信息是什么？
- 加载/处理过程中会发生什么？
- 是否有无障碍要求？
- 响应式行为的预期是什么？

### 步骤三：测试用例生成

**生成以下类别的测试用例：**

1. **功能测试用例**
   - 每个交互元素的功能
   - 数据输入和验证
   - 用户流程和导航
   - 业务逻辑执行
   - **区域功能区分**
     - **顶部导航区**：登录、设置、下载按钮等
     - **左侧导航区**：新对话、AI 创作、更多、历史对话、关于等
     - **主内容区**：欢迎信息、推荐问题、对话内容等
     - **底部输入区**：输入框、功能按钮、更多按钮等
   - **按钮功能区分**
     - 导航栏按钮（如"更多"、"设置"等）
     - 操作按钮（如"提交"、"搜索"等）
     - 功能按钮（如"PPT生成"、"图像生成"等）

2. **UI/视觉测试用例**
   
   **2.1 Logo 检查**
   - Logo 位置是否正确（居中、对齐方式）
   - Logo 尺寸是否符合设计要求
   - Logo 是否清晰可见
   - Logo 在不同分辨率下的显示
   
   **2.2 按钮检查**
   - 按钮位置和对齐方式
   - 按钮尺寸（宽度、高度）
   - 按钮颜色和样式
   - 按钮文字内容和字体
   - 按钮状态（默认、悬停、点击、禁用）
   
   **2.3 输入框/搜索框检查**
   - 输入框位置和对齐
   - 输入框尺寸和形状
   - 输入框边框样式
   - 占位符文字内容和样式
   - 输入框焦点状态样式
   
   **2.4 图片/图标检查**
   - 图片位置和对齐
   - 图片尺寸和比例
   - 图片清晰度和质量
   - 图标颜色和样式
   - 图片加载状态
   
   **2.5 文字/标签检查**
   - 文字内容准确性
   - 字体样式（字体、字号、粗细）
   - 文字颜色和对齐
   - 文字换行和截断
   - 标签和内容的对应关系
   
   **2.6 布局检查**
   - 页面整体布局结构
   - 元素间距和边距
   - 网格对齐
   - 响应式布局（不同屏幕尺寸）
   - 元素重叠检查
   
   **2.7 颜色和视觉检查**
   - 主色调和辅助色
   - 颜色对比度（无障碍）
   - 渐变和阴影效果
   - 高亮和选中状态颜色
   - 整体视觉一致性

3. **交互测试用例**
   - 点击/触摸行为
   - 悬停/聚焦状态
   - 键盘导航
   - 手势操作（移动端）

4. **边界情况和负面测试**
   - 空输入
   - 最大长度输入
   - 特殊字符
   - 并发操作

5. **兼容性测试用例**
   - 跨浏览器行为（Web端）
   - 屏幕尺寸变化
   - 设备方向（移动端）

## 输出格式

### 格式一：Markdown 格式（用于预览）

```
## 测试用例编号：[TC-XXX]
**测试用例名称：** [描述性名称]

**前置条件：** [设置要求]

**测试步骤：**
1. [步骤1]
2. [步骤2]
3. [步骤3]

**预期结果：** [应该发生什么]

**备注：** [附加信息、边界情况或需要澄清的问题]
```

### 格式二：Excel 表格格式（用于预览）

使用 Markdown 表格格式输出，可以直接复制到 Excel 中。

| 用例ID | 测试场景 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 备注 |
|--------|---------|---------|---------|---------|--------|------|
| TC-001 | 正常搜索功能 | 访问百度首页 | 1. 在搜索框中输入"测试"<br>2. 点击"百度一下"按钮 | 1. 搜索框接受输入<br>2. 跳转到搜索结果页面，显示与"测试"相关的搜索结果 | 高 | 无 |

**重要规则：**
- **测试步骤**和**预期结果**必须一一对应
- 如果有 3 条测试步骤，必须有 3 条预期结果
- 使用 `<br>` 换行来表示多条步骤或预期结果
- **优先级**可使用：高、中、低

### 格式三：直接生成 CSV 文件（Excel 可打开，推荐）

1. 将测试用例数据保存为 JSON 格式
2. 使用项目中的 `generate_csv.py` 脚本生成 CSV 文件

**JSON 数据格式：**
```json
[
  {
    "id": "TC-001",
    "scenario": "测试场景",
    "preconditions": "前置条件",
    "steps": "1. 步骤1\n2. 步骤2",
    "expected": "1. 预期1\n2. 预期2",
    "priority": "高",
    "notes": "备注"
  }
]
```

**生成 CSV 文件的步骤：**
1. 使用 `Write` 工具将测试用例保存为 `test_cases.json` 文件
2. 运行命令：`python generate_csv.py test_cases.json 测试用例.csv`
3. CSV 文件将生成在当前目录下
4. 使用 Excel 打开生成的 CSV 文件即可

## 示例

### 示例一：登录页面测试用例（Excel 表格格式）

当用户提供登录页面 UI 时：

1. **分析元素：** 用户名输入框、密码输入框、登录按钮、忘记密码链接、记住我复选框
2. **推断需求：**
   - 字段可能是必填的
   - 密码应该被掩码
   - 需要验证消息
   - 记住我功能
3. **生成测试用例（Excel 表格格式）：**

| 用例ID | 测试场景 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 备注 |
|--------|---------|---------|---------|---------|--------|------|
| TC-001 | 有效登录流程 | 访问登录页面 | 1. 在用户名输入框输入有效用户名<br>2. 在密码输入框输入有效密码<br>3. 点击登录按钮 | 1. 用户名输入框接受输入<br>2. 密码输入框接受输入并掩码显示<br>3. 成功登录，跳转到首页 | 高 | 无 |
| TC-002 | 空用户名验证 | 访问登录页面 | 1. 不输入用户名<br>2. 输入有效密码<br>3. 点击登录按钮 | 1. 用户名为空时焦点移出应显示必填提示<br>2. 点击登录按钮后显示用户名必填错误提示<br>3. 不进行登录提交 | 高 | 无 |
| TC-003 | 记住我功能 | 访问登录页面 | 1. 勾选"记住我"复选框<br>2. 输入有效用户名和密码<br>3. 点击登录按钮<br>4. 关闭浏览器，重新访问登录页面 | 1. "记住我"复选框可勾选/取消<br>2. 成功登录<br>3. 重新访问时用户名自动填充 | 中 | 无 |

    "notes": "无"
  },
  {
    "id": "TC-002",
    "scenario": "空用户名验证",

- 双击生成的 `.csv` 文件，Excel 会自动打开

## 重要指南

- **深入理解需求，不要表面分析**
  - 分析每个功能点的业务逻辑和用户意图
  - 理解用户使用场景和操作流程
  - 识别隐性需求和边界条件
  - 不要只看表面描述，要思考用户实际操作

- **按点拆分用例，禁止合并**
  - 每个独立的测试点必须单独拆分成一个用例
  - 一个用例只验证一个明确的测试点
  - 禁止将多个测试点合并到同一个用例中
  - 例如："登录功能"不是一个用例，应该拆分为：有效登录、密码错误、用户名为空、密码为空、用户不存在等多个用例

- **用例必须完整**
  - 步骤必须清晰具体，可操作执行
  - 预期结果必须与步骤一一对应
  - 每条步骤对应一条预期结果
  - 禁止在steps或expected中写多个要点合并的描述

- **拆分标准**
  - 每个输入字段的验证为一个独立用例
  - 每个按钮的每种状态为一个独立用例
  - 每个业务流程节点为一个独立用例
  - 每个边界条件为一个独立用例

- **始终对模糊的 UI 元素提出澄清问题**
- **在推断需求时记录假设**
- **按风险和使用频率对测试用例进行优先级排序**
- **包括正面和负面测试场景**
- **注意任何需要需求澄清的 UI 元素**

## 需求理解示例

**需求描述：** "用户可以登录系统"

**深入理解后应识别：**
1. 用户名输入框 → 必填验证、格式验证、长度验证、中英文支持
2. 密码输入框 → 必填验证、长度验证、可见/隐藏切换
3. 登录按钮 → 点击触发验证、加载状态、重复点击防抖
4. 记住我复选框 → 勾选/取消、保存登录状态
5. 忘记密码链接 → 点击跳转、链接有效性
6. 用户不存在 → 错误提示内容
7. 密码错误 → 错误提示内容、错误次数限制
8. 登录成功 → 跳转页面、显示用户信息
9. 连续快速点击 → 防重复提交
