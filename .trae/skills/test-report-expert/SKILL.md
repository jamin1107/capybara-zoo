# 测试报告生成专家

这个技能帮助测试工程师生成专业、规范、可直接用于项目交付的软件测试报告。根据用户提供的项目信息和测试数据，自动分析问题原因、给出优化建议，并生成正式的上线评估结论。

## 何时使用这个 Skill

**在以下情况下调用：**
- 用户需要生成软件测试报告
- 用户询问如何生成测试报告
- 用户提供了测试数据并要求生成报告
- 用户提到"测试报告"、"测试总结"、"上线评估"等关键词

## 工作流程

### 步骤一：信息收集

**需要收集的项目信息：**

1. **项目名称** - 被测系统或项目的名称
2. **测试版本** - 本次测试的版本号
3. **测试周期** - 测试开始和结束日期
4. **测试环境** - 包括：
   - 操作系统（Windows/Mac/Linux）
   - 浏览器类型及版本（Chrome/Firefox/Safari等）
   - 服务器环境（开发环境/测试环境/预发布环境）
   - 数据库版本
   - 其他相关环境信息
5. **测试人员** - 执行测试的人员姓名
6. **测试类型** - 功能测试 / 接口测试 / 兼容性测试 / 性能测试 / 回归测试
7. **测试范围与模块** - 本次测试覆盖的功能模块
8. **用例执行数据** - 总用例数、通过数、失败数、阻塞数、未执行数
9. **缺陷数据** - 总缺陷数、致命/严重/一般/轻微数量
10. **遗留问题** - 尚未解决的问题
11. **测试结论** - 是否允许上线（如用户未提供，根据数据自动分析）

### 步骤二：数据分析

**自动分析内容：**
- 用例通过率计算及分析
- 缺陷分布及严重程度分析
- 测试覆盖率评估
- 风险识别
- 问题根因推测

### 步骤三：报告生成

**生成标准测试报告，包含以下章节：**

1. **概述**
   - 项目背景
   - 测试目的
   - 测试范围摘要

2. **测试范围**
   - 测试模块详细列表
   - 功能点覆盖说明
   - 未测试范围说明

3. **测试环境**
   - 硬件环境
   - 软件环境
   - 网络环境
   - 测试数据准备

4. **用例执行情况**
   - 用例执行统计表格
   - 通过率图表说明
   - 失败用例简要分析

5. **缺陷统计**
   - 缺陷汇总表格
   - 按严重程度分布
   - 按模块分布
   - 缺陷趋势分析
   - 典型缺陷示例

6. **风险与遗留问题**
   - 已识别风险清单
   - 风险等级评估
   - 遗留问题列表
   - 影响分析

7. **优化建议**
   - 针对缺陷的改进建议
   - 测试流程优化建议
   - 质量提升建议

8. **测试结论**
   - 质量评估
   - 上线建议（允许上线/有条件上线/不建议上线）
   - 后续工作建议

## 使用示例

### 示例一：完整数据生成报告

当用户提供所有11项信息时，直接生成完整的测试报告。

**用户输入：**
```
1. 项目名称：电商平台系统
2. 测试版本：v2.1.0
3. 测试周期：2024-01-15 至 2024-01-20
4. 测试环境：
   - 操作系统：Windows 10 / macOS 14
   - 浏览器：Chrome 120.0 / Firefox 121.0
   - 测试环境：预发布环境
5. 测试人员：张三、李四
6. 测试类型：功能测试、回归测试
7. 测试范围与模块：用户登录、商品浏览、购物车、订单管理、支付功能
8. 用例执行数据：总用例数150、通过135、失败10、阻塞5、未执行0
9. 缺陷数据：总缺陷数15、致命1、严重3、一般8、轻微3
10. 遗留问题：支付功能在Safari浏览器下偶现失败，需进一步排查
11. 测试结论：待分析
```

**输出：**
根据以上数据生成完整的Markdown格式测试报告，包含数据分析、问题原因分析、优化建议和上线评估结论。

### 示例二：部分数据生成报告

当用户提供部分信息时，先询问缺失的信息，然后生成报告。

**用户输入：**
```
项目名称：OA办公系统
测试版本：v1.5.0
用例执行：通过120，失败8，总130
缺陷：致命0，严重2，一般10，轻微5
```

**输出：**
先询问缺失信息（测试周期、测试环境、测试人员、测试类型、测试范围、遗留问题等），收集完整后生成报告。

## 重要指南

- **报告语言**：使用中文正式、简洁的语言
- **数据分析**：根据数据自动计算通过率、缺陷率等关键指标
- **问题分析**：对失败用例和缺陷进行深入分析，推测根因
- **建议务实**：优化建议应具体、可操作
- **结论客观**：上线建议应基于数据，客观公正
- **格式规范**：严格遵循企业测试报告的标准格式
